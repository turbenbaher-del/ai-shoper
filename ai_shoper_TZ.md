## 1. ЧТО СТРОИМ

### 1.1 Продукт в одном абзаце

Telegram Mini App, который за 30-60 секунд находит лучшие товары на маркетплейсах (Ozon, Wildberries, Я.Маркет) по свободному запросу пользователя. Юзер говорит "робот-пылесос до 30к для кота с длинной шерстью" — приложение читает отзывы, отсекает накрутки, сравнивает цены и показывает топ-3 варианта со ссылками на покупку.

### 1.2 Аудитория

- **Ядро:** 25-35 лет, активные пользователи Telegram, покупают на маркетплейсах 4-15 раз в месяц
- **Вторично:** 35-45 лет, ценят экономию времени, готовы платить за качество выбора
- **География:** Россия, базовый язык — русский
- **Платформа:** в основном мобильные устройства (95% трафика)

### 1.3 Бизнес-модель

Два потока выручки:

**CPA-партнёрки** (основной поток первые 3 месяца)
- Ozon Партнёры: 2-7% от чека
- WB через CPA-сети (Admitad/EPN): 1.5-5%
- Я.Маркет Партнёры: 2-10%
- Средний чек ~3-5к рублей, средний CPA ~50₽ на бесплатного юзера

**Premium-подписка** (растёт с третьего месяца)
- Месяц: 590₽
- Год: 4900₽ (≈410₽/мес, скидка 31%)
- 7 дней бесплатно
- Лимит free: 3 запроса/месяц, без price alerts, без истории

**Целевые показатели к месяцу 12:**
- 90 000 юзеров, 8 000 Premium
- ~10.2M ₽/мес оборот
- ~5-6M ₽/мес чистыми

### 1.4 Что НЕ делаем

- **Не покупаем за юзера** — только показываем варианты со ссылками
- **Не работаем как маркетплейс** — нет корзины, оплаты, доставки
- **Не поддерживаем категории с риском:** одежда (размеры), продукты (срок), мебель (доставка), лекарства (лицензирование)
- **Не отображаем рекламу** — выручка только CPA + Premium
- **Не собираем избыточные данные** — минимум для работы

## 2. ФУНКЦИОНАЛЬНОСТЬ MVP

### 2.1 Скоуп MVP (что должно работать к запуску)

**Поддерживаемые категории (5 на старте):**
1. Робот-пылесос
2. Наушники
3. Зимняя куртка
4. Кофемашина
5. Смарт-часы

**Поддерживаемые маркетплейсы:**
- Ozon (через Партнёрку)
- Wildberries (через CPA-сеть Admitad)
- Я.Маркет (через Партнёрку)

**8 экранов P1 (must для MVP):**

| # | Экран | Главное действие |
|---|-------|-----------------|
| 1 | Welcome | Знакомство, кнопка "Начать" |
| 2 | Quiz (4 шага) | Сбор контекста о юзере |
| 3 | Home | Главный экран, кнопка "Что найти?" |
| 4 | Input | Ввод запроса (голос/текст) |
| 5 | Processing | Анимация поиска 5 этапов |
| 6 | Results | Топ-3 варианта |
| 7 | Soft Paywall | После 3-го запроса |
| 8 | Tracked items | Отслеживание цен |

Подробности см. в `ai_shoper_prototype.html` — там все экраны кликабельны.

### 2.2 Главный пользовательский сценарий

```
Юзер запускает бота в Telegram
   ↓
Welcome → клик "Начать"
   ↓
Quiz 4 шага (для кого, маркетплейсы, приоритет, категории)
   ↓
Home (главный экран с кнопкой "Что найти?")
   ↓
Input → юзер описывает что нужно (голос или текст)
   ↓
Processing (8-15 сек, видны этапы: Понимаю → Сканирую Ozon → ...)
   ↓
Results → топ-3 варианта с ценами по 3 маркетплейсам
   ↓
   ├─→ "Купить" → переход на маркетплейс с CPA-ссылкой
   ├─→ "Поделиться" → шер-карточка в TG (виральность)
   └─→ "Следить" → push когда подешевеет (retention)
```

### 2.3 Soft Paywall механика

**3 бесплатных запроса в месяц для каждого юзера.**

После 3-го успешного запроса (т.е. дошедшего до Results) показываем Paywall.

- Можно закрыть и продолжать пользоваться tracked items / историей
- Нельзя сделать 4-й запрос пока не подпишешься
- Trial: 7 дней бесплатно, требует привязки карты через ЮKassa или Telegram Stars
- Reset бесплатных запросов: 1-го числа каждого месяца

**Дополнительные contextual paywalls:**
- Добавление в tracked items сверх 3-х → "Premium даёт безлимит"
- Доступ к истории старше 7 дней → "Premium хранит историю навсегда"

### 2.4 Wizard of Oz режим (первые 50 запросов)

Прежде чем подключать реальные API маркетплейсов, первые 50 запросов обрабатываем вручную:

1. Юзер пишет запрос в боте
2. Бот отвечает: "Принял, найду варианты в течение 5-10 минут"
3. Владелец проекта получает уведомление о новом запросе
4. Запускает `python test_run.py` с этим запросом локально
5. Берёт JSON-результат и форматирует как сообщение в TG
6. Отправляет юзеру через бота

**Этот режим даёт:**
- 50 живых наблюдений за реальными запросами
- Понимание где парсинг плохой, какие запросы вне категорий
- Время на подключение реальных API маркетплейсов параллельно

## 3. ДИЗАЙН-ПРИНЦИПЫ

### 3.1 Архетип бренда: "Знающий друг"

- Знает рынок лучше тебя, но не выпендривается
- Говорит на твоём языке, без панибратства
- Прямой, но не грубый
- На "ты", с уважением

### 3.2 Tone of voice

**Делаем:**
- Прямо: "Нашёл 3 варианта"
- Конкретно: "Сэкономишь 2 800₽"
- Без украшательств: "Этот лучший, бери его"
- На "ты", дружелюбно

**Не делаем:**
- "Превосходный", "идеальный", "обязательно понравится"
- Слишком много восклицаний и эмодзи
- Маркетинговые клише ("инновационный", "передовой")
- Технический жаргон ("ИИ-модель обрабатывает запрос")

### 3.3 Палитра (полный список см. в `ai_shoper_design_spec.md`)

- Фон: `#0A0E13` (тёмный)
- Поверхности: `#161B22`
- Текст: `#E6EDF3`
- Акцент (главный бренд-цвет): `#00D9A6` (бирюзово-зелёный)
- Семантические: success `#3FB950`, warning `#D29922`, danger `#F85149`, info `#58A6FF`

### 3.4 Типографика

- Один шрифт: Inter (Google Fonts)
- 6 размеров: Display 32 / H1 24 / H2 18 / BodyL 16 / Body 14 / Caption 12 / Tiny 11
- Для цен: `font-variant-numeric: tabular-nums`

### 3.5 Анимации

- Длительность: 200-400мс
- Easing: `cubic-bezier(0.16, 1, 0.3, 1)` — единственный в проекте
- Уважаем `prefers-reduced-motion`
- НЕ используем: confetti, sparkles, bounce-эффекты, parallax

### 4.4 Модель данных (PostgreSQL)

```sql
-- Пользователи
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    tg_user_id BIGINT UNIQUE NOT NULL,
    tg_username TEXT,
    tg_first_name TEXT,
    tg_language_code TEXT DEFAULT 'ru',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    last_active_at TIMESTAMPTZ DEFAULT NOW(),

    -- Из quiz онбординга
    quiz_completed BOOLEAN DEFAULT FALSE,
    quiz_data JSONB,  -- {who: 'для семьи', marketplaces: [...], priority: 'цена'}

    -- Лимиты
    free_searches_used INTEGER DEFAULT 0,
    free_searches_reset_at TIMESTAMPTZ,

    -- Premium
    is_premium BOOLEAN DEFAULT FALSE,
    premium_until TIMESTAMPTZ,
    premium_plan TEXT,  -- 'month' or 'year' or 'trial'

    -- Согласия
    pdn_consent_at TIMESTAMPTZ,
    push_consent BOOLEAN DEFAULT FALSE
);

-- Поисковые запросы (история)
CREATE TABLE queries (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ DEFAULT NOW(),

    -- Что искали
    raw_text TEXT NOT NULL,
    parsed_request JSONB NOT NULL,

    -- Результаты
    products JSONB NOT NULL,           -- top_3
    total_cost_rub NUMERIC(8, 2),
    processing_time_seconds NUMERIC(5, 1),

    -- Аналитика
    is_clarification BOOLEAN DEFAULT FALSE,  -- если был ответ "уточни"
    clicked_buy BOOLEAN DEFAULT FALSE,
    clicked_share BOOLEAN DEFAULT FALSE,
    cpa_tracking_id TEXT  -- для атрибуции выручки
);

CREATE INDEX idx_queries_user_id_created ON queries(user_id, created_at DESC);

-- Отслеживание цен
CREATE TABLE tracked_items (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ DEFAULT NOW(),

    -- Товар
    marketplace TEXT NOT NULL,
    sku TEXT NOT NULL,
    name TEXT NOT NULL,
    url TEXT NOT NULL,
    image_url TEXT,

    -- Цены
    initial_price INTEGER NOT NULL,    -- цена когда поставили на отслеживание
    current_price INTEGER NOT NULL,
    last_checked_at TIMESTAMPTZ DEFAULT NOW(),

    -- Алерты
    alert_threshold_pct INTEGER DEFAULT 5,  -- срабатывает при -5% и более
    last_alert_sent_at TIMESTAMPTZ,

    is_active BOOLEAN DEFAULT TRUE
);

CREATE INDEX idx_tracked_user_active ON tracked_items(user_id, is_active);
CREATE INDEX idx_tracked_check ON tracked_items(last_checked_at) WHERE is_active = TRUE;

-- История цен (для графиков)
CREATE TABLE price_history (
    id BIGSERIAL PRIMARY KEY,
    tracked_item_id BIGINT REFERENCES tracked_items(id) ON DELETE CASCADE,
    price INTEGER NOT NULL,
    captured_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_price_history_item ON price_history(tracked_item_id, captured_at);

-- Подписки
CREATE TABLE subscriptions (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id) ON DELETE CASCADE,

    plan TEXT NOT NULL,                  -- 'month', 'year', 'trial'
    status TEXT NOT NULL,                -- 'active', 'cancelled', 'expired', 'pending'

    started_at TIMESTAMPTZ DEFAULT NOW(),
    expires_at TIMESTAMPTZ NOT NULL,
    cancelled_at TIMESTAMPTZ,

    -- Платёж
    payment_provider TEXT,               -- 'yukassa', 'tg_stars'
    payment_id TEXT,
    amount_rub NUMERIC(8, 2),

    auto_renew BOOLEAN DEFAULT TRUE
);

CREATE INDEX idx_subs_user ON subscriptions(user_id, status);
```

### 4.5 HTTP API эндпойнты

Все эндпойнты под `/api/v1`. Авторизация через Telegram WebApp initData (валидируется на сервере).

#### Auth
- `POST /api/v1/auth/telegram` — авторизация по initData, возвращает session token
  - Body: `{init_data: "..."}` (raw initData строка из TG)
  - Response: `{user: {...}, token: "...", is_new: true/false}`

#### Quiz
- `POST /api/v1/quiz` — сохранение результатов онбординга
  - Body: `{who, marketplaces, priority, categories}`
  - Response: `{ok: true}`

#### Search
- `POST /api/v1/search` — главный эндпойнт поиска
  - Body: `{query: "робот-пылесос до 30к для кота"}`
  - Response (SSE стрим или обычный JSON):
    ```json
    {
      "query": "...",
      "title": "Нашёл 3 варианта",
      "subtitle": "Прочитал 187 отзывов · сравнил 3 маркетплейса · отсёк 12 накруток",
      "products": [...],
      "share_text": "...",
      "processing_time_seconds": 12.4,
      "needs_clarification": false
    }
    ```
  - Если `needs_clarification`: `{needs_clarification: true, clarification: "Какой бюджет?"}`
  - При исчерпании лимита: 402 Payment Required + `{paywall: true}`

- `GET /api/v1/search/history?limit=20` — история запросов
- `GET /api/v1/search/{id}` — детали конкретного запроса

#### Tracked
- `GET /api/v1/tracked` — список отслеживаемых товаров
- `POST /api/v1/tracked` — добавить товар
  - Body: `{marketplace, sku, name, url, image_url, initial_price}`
- `DELETE /api/v1/tracked/{id}` — снять с отслеживания
- `GET /api/v1/tracked/{id}/history` — история цен для графика

#### Subscription
- `GET /api/v1/subscription` — статус подписки
- `POST /api/v1/subscription/start` — начать trial или оплатить
  - Body: `{plan: "month" | "year" | "trial"}`
  - Response: `{redirect_url: "https://yukassa..."}` или `{ok: true}` для trial
- `POST /api/v1/subscription/cancel` — отменить подписку

#### Webhook
- `POST /api/v1/webhook/yukassa` — нотификации от ЮKassa о статусе платежа

#### CPA tracking
- `GET /api/v1/click/{query_id}/{product_idx}` — редирект на маркетплейс с CPA-тегом, логирует клик

## 7. ВЫБОР МОДЕЛИ CLAUDE ДЛЯ РАЗРАБОТКИ

**Этот раздел — для Claude Code и владельца проекта. Определяет какую модель Anthropic использовать на разных этапах работы для оптимального баланса качество/цена/скорость.**

### 7.1 Общий принцип: смешанная стратегия

Не "только Sonnet" и не "только Opus", а **роутинг по типу задачи**:

- **Sonnet 4.6** (`claude-sonnet-4-6`) — default, 80% задач
- **Opus 4.7** (`claude-opus-4-7`) — для сложных задач, 20% задач

**Цены (на май 2026):**
- Sonnet: $3 / $15 за 1M input/output токенов
- Opus: $5 / $25 за 1M токенов (в 1.7× дороже)

**Бенчмарки (по независимым тестам):**
- Sonnet на SWE-bench Verified: ~79.6%
- Opus 4.7 на SWE-bench Verified: ~87.6%
- На CursorBench (реальные задачи в IDE) Opus 4.7 решает на 12 пунктов больше задач чем Opus 4.6
- На 93-задачном тесте Hex Opus 4.7 решил 4 задачи, которые ни Opus 4.6, ни Sonnet 4.6 не смогли

**Логика выбора:**
- Sonnet хватает для большинства production-задач: CRUD-эндпойнты, React компоненты по готовым стилям, миграции БД, интеграции по понятным API
- Opus берёшь когда задача многошаговая, требует держать большой контекст или цена ошибки высока (деньги, безопасность)
- Sonnet быстрее — для прямолинейного кода это важнее качества "на 5% выше"

### 7.2 Распределение по задачам проекта

Таблица соответствия задач из плана реализации (раздел 5) и рекомендованной модели:

| Категория задач | Модель | Обоснование |
|------------------|--------|-------------|
| Структура папок, базовый FastAPI, Docker setup | **Sonnet** | Шаблонная работа, типовой код |
| SQLAlchemy модели, Alembic миграции | **Sonnet** | Стандартный паттерн |
| API эндпойнты (auth, tracked, history, subscription) | **Sonnet** | CRUD-логика, прямолинейная |
| React компоненты по дизайн-системе | **Sonnet** | Перенос готовых стилей из HTML |
| Экраны Mini App (Welcome, Quiz, Home, Input) | **Sonnet** | Готовый прототип, надо перенести |
| Bottom sheet, Empty states, базовые UI | **Sonnet** | Простая логика |
| Скрипты, тесты, e2e сценарии | **Sonnet** | Шаблонная работа |
| **Доработка ИИ-пайплайна и промптов** | **Opus** | 6 этапов, надо держать контекст всего pipeline |
| **Адаптеры маркетплейсов** (Ozon/WB/Я.Маркет API) | **Opus** | Внешние API сложные, документация часто кривая |
| **Celery воркеры** (price tracker + alerts) | **Opus** | Многопоточность, гонки состояний |
| **Биллинг ЮKassa + webhook handling** | **Opus** | Деньги, идемпотентность критична |
| **Soft paywall логика** (счётчики, reset, лимиты) | **Opus** | Граничные случаи, нужна аккуратность |
| **Дебаг сложных багов** | **Opus** | Когда Sonnet 2 раза подряд не справился |
| **Архитектурные решения по ходу** | **Opus** | Дорогие ошибки в основах |
| **Code review** критичных модулей перед запуском | **Opus** | Свежий взгляд экспертного уровня |

### 7.3 Как переключать модель в Claude Code

В Claude Code модель выбирается через флаг или команду в сессии. Конкретный синтаксис зависит от версии — см. документацию.

**Практическая стратегия:**
1. **Начинать каждую сессию с Sonnet** как default
2. **Переключаться на Opus** при попадании на задачу из колонки "Opus" в таблице 7.2
3. **Возвращаться на Sonnet** когда сложная часть закрыта

**Альтернативный паттерн — "Opus планирует, Sonnet исполняет":**
- Попроси Opus спланировать неделю работ или сложную фичу
- Получи от него детальный план задач, подводных камней, edge cases
- Выполняй план с Sonnet — он быстрее и достаточно умён для исполнения

### 7.4 Сигналы что пора переключиться на Opus

Симптомы говорящие "Sonnet уже не справляется, бери Opus":

- **2 раза подряд** Sonnet предложил код который не работает или не учитывает требования
- **Sonnet "забывает"** части задачи в длинном диалоге (контекст переполняется)
- Задача требует **держать в голове >5 связанных файлов** одновременно
- **Гонки состояний, дедлоки, race conditions** в фоновых задачах
- Любая работа с **деньгами или безопасностью** где ошибка дорогая
- **Анализ внешнего API** с плохой документацией (например, неофициальные эндпойнты WB)

### 7.5 Грубая прикидка стоимости

При плотной работе соло-фаундера в Claude Code (~50-100к токенов в день):

- **Только Sonnet:** ~$300-500 за 6 недель MVP
- **Только Opus:** ~$500-900 за 6 недель MVP
- **Рекомендуемая 80/20:** ~$350-550 за 6 недель MVP

Разница между крайностями ~$300-400 за весь MVP. Если бюджет совсем жмёт — старт только на Sonnet, переключайся на Opus только когда упёрся.

### 7.6 Контекст важнее модели

**Самое большое влияние на качество кода — НЕ выбор модели, а качество контекста:**

- Передавай Claude Code этот ТЗ целиком в начале каждой сессии
- Передавай готовые материалы: прототип, дизайн-систему, пайплайн
- Не давай модели "догадаться" — пиши явно
- Любая модель с хорошим контекстом работает лучше чем Opus без контекста

Это правило важнее любых рекомендаций по выбору модели в этом разделе.
