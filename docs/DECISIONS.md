# Лог решений (DECISIONS.md)

Все отклонения от ТЗ и архитектурные решения фиксируются здесь с датой и причиной.

---

## 2026-05-19

### Session management — упрощённый in-memory store
**Решение:** Сессионные токены хранятся в `_fake_sessions` dict в `api/deps.py` вместо Redis.  
**Причина:** Быстрый старт для MVP. Redis сессии имеют смысл когда появится горизонтальное масштабирование.  
**Когда менять:** Перед запуском на продакшен с несколькими workers — перенести в Redis.

### Модели Claude для пайплайна
**Решение:** Парсинг запроса (шаг 1) и форматирование (шаг 6) — Sonnet. Анализ отзывов (шаг 4) и ранжирование (шаг 5) — Opus.  
**Причина:** Соответствует разделу 7.2 ТЗ. Шаги с накрутками и финансовой аккуратностью — Opus.

### Marketplace адаптеры — реальные API + fallback на mock
**Решение:** Адаптеры используют реальные эндпойнты:
  - WB: публичный `search.wb.ru/exactmatch/...` (работает без ключа)
  - Ozon: неофициальный `composer-api.bx` (использует фронт сайта)
  - Я.Маркет: Partner API при наличии OAuth, иначе mock
При сбое API или отсутствии ключей — возвращаются mock-данные (для Wizard of Oz режима).  
**Причина:** Реальные данные нужны для качества ранжирования, но MVP должен работать и без партнёрских подключений.  
**Риски:** WB и Ozon endpoints — неофициальные, могут сломаться. Структуру композер-API Ozon парсим оборонительно с fallback на mock при изменениях.

### Per-host rate limiting для маркетплейсов
**Решение:** Общий HTTP-клиент `marketplaces/_http.py` с `HostLimit` на каждый хост (concurrency + min_interval).  
**Причина:** Маркетплейсы банят при слишком частых запросах. Лимиты подобраны эмпирически с запасом.  
**Когда менять:** Если начнём упираться в 429 — снижать concurrency/повышать interval.

### Идемпотентность алертов о цене — атомарный UPDATE с WHERE
**Решение:** В `price_tracker._process_item` помечаем `last_alert_sent_at` через `UPDATE … WHERE last_alert_sent_at < cooldown_cutoff`. Если строка не обновилась — другой воркер уже отметил, алерт не дублируется.  
**Причина:** Простая дедупликация без distributed locks. Trade-off: при сбое отправки в TG между UPDATE и `delay()` — алерт пропадёт, но пользователь получит его на следующей итерации (через 1 час).

### YuKassa webhook — UPSERT по payment_id
**Решение:** Webhook UPSERT-ит подписку по `payment_id`. Если webhook пришёл раньше чем `create_yukassa_payment` успел сохранить запись (race) — webhook создаёт подписку сам.  
**Причина:** Производительность ЮKassa может опередить локальный commit. Полагаемся на UNIQUE(payment_id) и идемпотентность операции активации премиума.

### Prompt caching для system-промптов Claude
**Решение:** В `call_claude_json` через `cache_system=True` помечаем system как `ephemeral` cache_control.  
**Причина:** Anthropic кеширует system до 5 минут → повторные запросы в пайплайне (тем же воркером в течение сессии) экономят до 90% на input tokens.  
**Когда отключать:** Если system короче ~1024 токенов — кеширование не окупается.

### React роутинг — state machine вместо react-router
**Решение:** Переключение экранов через Zustand store (`screen` поле) без URL-роутинга.  
**Причина:** Telegram Mini App не использует браузерную историю. State machine проще и предсказуемее.

### Auth token — Bearer header
**Решение:** Простой Bearer token в заголовке. Не JWT.  
**Причина:** Для MVP достаточно. JWT добавит сложность верификации подписи.  
**Когда менять:** При переносе на несколько инстансов (stateless auth).
